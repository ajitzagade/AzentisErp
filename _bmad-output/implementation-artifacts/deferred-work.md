## Deferred from: code review of Epic 2 (2026-07-11)

**Process note**: this review used three parallel reviewers given real Bash access to the running bench, not a diff — Epic 2's actual changes live entirely in gitignored `myerp/sites/` (config flag, new site directories, Tenant Settings data), so there was no source diff to construct. All three reviewers independently re-ran or extended the stories' own verification commands against the live system rather than trusting the transcribed output, which is how several of the findings below were caught.

- **MUST-DO for any production deployment (not optional, not "nice to have"): Nginx must strip/overwrite client-supplied `X-Frappe-Site-Name` headers before proxying to gunicorn.** `frappe/app.py::init_request()` resolves the target site as `_site or request.headers.get("X-Frappe-Site-Name") or get_site_name(request.host)` — the `X-Frappe-Site-Name` header takes priority over the `Host` header. Confirmed live: `curl -H "Host: dev.local" -H "X-Frappe-Site-Name: priya.dev.local" .../get_branding` returns Priya's tenant data under `dev.local`'s hostname; works in both directions between any two sites on the bench. This is Frappe core behavior (`frappe/app.py`) — AD-1 forbids modifying it, and there is no `our_brand` hook that can intercept this early enough (`init_request()` has already resolved the site and connected to its DB before any `before_request` hook runs). **The only correct fix is at the reverse-proxy layer**: production Nginx configs must explicitly set/overwrite `X-Frappe-Site-Name` themselves (a standard Frappe production-hardening step) so a client-supplied value can never reach gunicorn. Actionable at Epic 4 (production server setup, AD-7/AD-11's pipeline) — this must be part of the production Nginx config from day one, not a follow-up hardening pass, since without it every tenant's isolation is one crafted HTTP header away from being bypassed by anyone who can reach the server at all.
- **Case-sensitivity portability gap between dev (macOS) and likely production (Linux) filesystems, unverified.** `frappe/utils/__init__.py::get_site_name()` does no case normalization (`hostname.split(":", 1)[0]`), and Frappe's site lookup is a literal folder-name match under `sites/`. On this dev machine (macOS/APFS, case-insensitive by default), `curl -H "Host: DEV.LOCAL"` and `curl -H "Host: Priya.Dev.Local"` both resolved correctly — but this only works because the filesystem itself is forgiving, not because Frappe normalizes case. A case-variant tenant hostname (e.g. a DNS provider or client typo introducing different casing) would very likely 404 on a case-sensitive Linux production filesystem where this dev environment's tests can't catch it. Actionable at Epic 4 — worth an explicit production-environment smoke test with a deliberately case-varied hostname before relying on this dev environment's routing tests as sufficient.
- **Tenant logo/splash images are not actually per-tenant** — `install.py::after_install()` sets `Website Settings.app_logo`/`splash_image` to the single shared, symlinked `/assets/our_brand/images/logo.png` on every site, regardless of `Tenant Settings.logo` (a field that exists on the DocType — Story 1.6 — but is never read by any code). Every tenant's logo and splash screen currently show the same platform logo, only colors differ per tenant. This connects to Epic 1's already-deferred "logo/favicon fields collected but never rendered" item (now more concretely confirmed, since two real tenant sites exist and both show the identical logo image). Actionable whenever a dedicated per-tenant logo story is scheduled (Epic 2/3 territory per Epic 1's original framing) — needs real per-site file upload handling (`Website Settings.app_logo` would need to point at a `/private/files/...` upload specific to that site, not the shared app-package asset), not just reading the existing `Tenant Settings.logo` field's URL naively.
- **`serve_default_site: true` + no `currentsite.txt` is a currently-dormant risk, not an active one — worth a guard before it becomes active.** If `bench use <site>` is ever run in this bench (creating `sites/currentsite.txt`), `serve_default_site: true` combined with a set default site could make an unmatched/misconfigured tenant hostname silently serve that default site's content instead of 404ing — directly contradicting Story 2.1 AC2's "no fallback to a default/wrong site." Currently inactive only because `currentsite.txt` doesn't exist and no default has ever been set. Actionable at Epic 4 — either explicitly set `serve_default_site: false` as part of production hardening, or ensure the provisioning pipeline never runs `bench use`.
- **Path-traversal / malformed `Host` header handling in Frappe core is untested and unmitigated at the application layer.** Edge Case Hunter's static-analysis finding (not independently reproduced in this pass): a `Host` header containing path-traversal characters could reach `get_site_name()`/filesystem path construction without validation, potentially producing a 500 instead of the clean 404 confirmed for ordinary nonexistent hostnames. Frappe core again (AD-1 applies) — the same production-Nginx mitigation layer that must strip `X-Frappe-Site-Name` should also validate/reject malformed `Host` headers before they reach gunicorn. Actionable alongside the `X-Frappe-Site-Name` fix at Epic 4 — worth an explicit reproduction test against the real Nginx config once it exists, not just this dev server.
- **`Tenant Settings.site_name`/`custom_domain` fields (Story 1.6) are never populated by any code.** AD-11's identity model intends these to mirror the site's canonical internal name/eventual public domain, but nothing in `install.py` or elsewhere writes them — both real tenant sites (`priya.dev.local`, `acme.dev.local`) have them empty. AC2 of Story 2.2 is satisfied via the bench's own folder-level site identity, not via these DocType fields. Actionable whenever Epic 4's provisioning pipeline is built (AD-7) — that pipeline should populate `site_name` at site-creation time as one of its steps.
- **No raw command/response artifacts are preserved for any story's verification evidence** — every curl/console check across this entire project is recorded as a hand-transcribed one-line summary, not a saved log or full response. This makes independent re-verification of historical claims impossible without re-running the commands fresh. Not fixing retroactively (would mean re-doing every prior story's verification), but worth adopting going forward if verification rigor becomes a recurring point of scrutiny: redirect key verification commands to a saved log file referenced from the story, at least for security- or isolation-relevant claims.
- **Redis cache/queue isolation across tenants is real but was never established by any story until this review** — `frappe.utils.redis_wrapper.RedisWrapper.make_key()` prefixes every key with `frappe.conf.db_name` (confirmed by reading source), so the bench-wide shared `redis_cache`/`redis_queue`/`redis_socketio` instances don't currently leak data between tenants despite not being per-site instances. This is a positive finding, not a gap — recorded here so it doesn't need re-investigating later, and so the "no cross-tenant bleed" story (2.3) has this surface documented rather than silently unexamined.
