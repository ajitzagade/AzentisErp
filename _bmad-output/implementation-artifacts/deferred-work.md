## Deferred from: code review of Epic 2 (2026-07-11)

**Process note**: this review used three parallel reviewers given real Bash access to the running bench, not a diff â€” Epic 2's actual changes live entirely in gitignored `myerp/sites/` (config flag, new site directories, Tenant Settings data), so there was no source diff to construct. All three reviewers independently re-ran or extended the stories' own verification commands against the live system rather than trusting the transcribed output, which is how several of the findings below were caught.

- **MUST-DO for any production deployment (not optional, not "nice to have"): Nginx must strip/overwrite client-supplied `X-Frappe-Site-Name` headers before proxying to gunicorn.** `frappe/app.py::init_request()` resolves the target site as `_site or request.headers.get("X-Frappe-Site-Name") or get_site_name(request.host)` â€” the `X-Frappe-Site-Name` header takes priority over the `Host` header. Confirmed live: `curl -H "Host: dev.local" -H "X-Frappe-Site-Name: priya.dev.local" .../get_branding` returns Priya's tenant data under `dev.local`'s hostname; works in both directions between any two sites on the bench. This is Frappe core behavior (`frappe/app.py`) â€” AD-1 forbids modifying it, and there is no `our_brand` hook that can intercept this early enough (`init_request()` has already resolved the site and connected to its DB before any `before_request` hook runs). **The only correct fix is at the reverse-proxy layer**: production Nginx configs must explicitly set/overwrite `X-Frappe-Site-Name` themselves (a standard Frappe production-hardening step) so a client-supplied value can never reach gunicorn. Actionable at Epic 4 (production server setup, AD-7/AD-11's pipeline) â€” this must be part of the production Nginx config from day one, not a follow-up hardening pass, since without it every tenant's isolation is one crafted HTTP header away from being bypassed by anyone who can reach the server at all.
- **Case-sensitivity portability gap between dev (macOS) and likely production (Linux) filesystems, unverified.** `frappe/utils/__init__.py::get_site_name()` does no case normalization (`hostname.split(":", 1)[0]`), and Frappe's site lookup is a literal folder-name match under `sites/`. On this dev machine (macOS/APFS, case-insensitive by default), `curl -H "Host: DEV.LOCAL"` and `curl -H "Host: Priya.Dev.Local"` both resolved correctly â€” but this only works because the filesystem itself is forgiving, not because Frappe normalizes case. A case-variant tenant hostname (e.g. a DNS provider or client typo introducing different casing) would very likely 404 on a case-sensitive Linux production filesystem where this dev environment's tests can't catch it. Actionable at Epic 4 â€” worth an explicit production-environment smoke test with a deliberately case-varied hostname before relying on this dev environment's routing tests as sufficient.
- **Tenant logo/splash images are not actually per-tenant** â€” `install.py::after_install()` sets `Website Settings.app_logo`/`splash_image` to the single shared, symlinked `/assets/our_brand/images/logo.png` on every site, regardless of `Tenant Settings.logo` (a field that exists on the DocType â€” Story 1.6 â€” but is never read by any code). Every tenant's logo and splash screen currently show the same platform logo, only colors differ per tenant. This connects to Epic 1's already-deferred "logo/favicon fields collected but never rendered" item (now more concretely confirmed, since two real tenant sites exist and both show the identical logo image). Actionable whenever a dedicated per-tenant logo story is scheduled (Epic 2/3 territory per Epic 1's original framing) â€” needs real per-site file upload handling (`Website Settings.app_logo` would need to point at a `/private/files/...` upload specific to that site, not the shared app-package asset), not just reading the existing `Tenant Settings.logo` field's URL naively.
- **`serve_default_site: true` + no `currentsite.txt` is a currently-dormant risk, not an active one â€” worth a guard before it becomes active.** If `bench use <site>` is ever run in this bench (creating `sites/currentsite.txt`), `serve_default_site: true` combined with a set default site could make an unmatched/misconfigured tenant hostname silently serve that default site's content instead of 404ing â€” directly contradicting Story 2.1 AC2's "no fallback to a default/wrong site." Currently inactive only because `currentsite.txt` doesn't exist and no default has ever been set. Actionable at Epic 4 â€” either explicitly set `serve_default_site: false` as part of production hardening, or ensure the provisioning pipeline never runs `bench use`.
- **Path-traversal / malformed `Host` header handling in Frappe core is untested and unmitigated at the application layer.** Edge Case Hunter's static-analysis finding (not independently reproduced in this pass): a `Host` header containing path-traversal characters could reach `get_site_name()`/filesystem path construction without validation, potentially producing a 500 instead of the clean 404 confirmed for ordinary nonexistent hostnames. Frappe core again (AD-1 applies) â€” the same production-Nginx mitigation layer that must strip `X-Frappe-Site-Name` should also validate/reject malformed `Host` headers before they reach gunicorn. Actionable alongside the `X-Frappe-Site-Name` fix at Epic 4 â€” worth an explicit reproduction test against the real Nginx config once it exists, not just this dev server.
- **`Tenant Settings.site_name`/`custom_domain` fields (Story 1.6) are never populated by any code.** AD-11's identity model intends these to mirror the site's canonical internal name/eventual public domain, but nothing in `install.py` or elsewhere writes them â€” both real tenant sites (`priya.dev.local`, `acme.dev.local`) have them empty. AC2 of Story 2.2 is satisfied via the bench's own folder-level site identity, not via these DocType fields. Actionable whenever Epic 4's provisioning pipeline is built (AD-7) â€” that pipeline should populate `site_name` at site-creation time as one of its steps.
- **No raw command/response artifacts are preserved for any story's verification evidence** â€” every curl/console check across this entire project is recorded as a hand-transcribed one-line summary, not a saved log or full response. This makes independent re-verification of historical claims impossible without re-running the commands fresh. Not fixing retroactively (would mean re-doing every prior story's verification), but worth adopting going forward if verification rigor becomes a recurring point of scrutiny: redirect key verification commands to a saved log file referenced from the story, at least for security- or isolation-relevant claims.
- **Redis cache/queue isolation across tenants is real but was never established by any story until this review** â€” `frappe.utils.redis_wrapper.RedisWrapper.make_key()` prefixes every key with `frappe.conf.db_name` (confirmed by reading source), so the bench-wide shared `redis_cache`/`redis_queue`/`redis_socketio` instances don't currently leak data between tenants despite not being per-site instances. This is a positive finding, not a gap â€” recorded here so it doesn't need re-investigating later, and so the "no cross-tenant bleed" story (2.3) has this surface documented rather than silently unexamined.

## Deferred from: code review of Epic 3 (2026-07-12)

**Process note**: three parallel reviewers ran against a correctly-constructed diff (untracked files included from the start this time, learning from Epic 1/2's diff-construction mistakes) plus direct access to the real bench. The most significant finding â€” `sync_blocked_modules()`'s write silently discarded for users with a Module Profile assigned â€” was independently reproduced by two reviewers with real empirical repro steps, not just theorized, and was patched (see Stories 3.1-3.5's consolidated Review Findings). Patched alongside it: `apply_preset()` was never callable outside a console session (now `@frappe.whitelist()`'d with a System Manager check), `MODULE_PRESETS["General"]` aliased rather than copied `TOGGLEABLE_MODULES` (now `list(...)`), `apply_preset()` on an unknown preset name raised a raw traceback (now a clean `frappe.throw`), `seed_module_presets()` could raise `DuplicateEntryError` on a concurrent run (now caught), and existing sites never got new/changed presets except by someone remembering to re-run `seed_module_presets()` by hand (now a `post_model_sync` patch runs it on every `bench migrate` â€” confirmed live: `priya.dev.local`, which had never been manually seeded, got all 4 presets automatically on the next migrate). AD-2's architecture text was also corrected to accurately describe Story 3.2's actual sidebar-filtering design (syncs to `User.block_modules` at save time, doesn't re-read Tenant Settings on every desk request) rather than leaving the original, now-inaccurate "goes through the cached accessor" claim standing.

- **No hook syncs a newly-created or newly-promoted System User's `block_modules`.** `sync_blocked_modules()` only runs on `Tenant Settings.on_update` â€” a staff member invited (or promoted from Website User to System User) after the tenant's last Tenant Settings save sees an unfiltered sidebar until some unrelated future Tenant Settings save happens to re-sync everyone. Actionable as a small addition: a `User` `after_insert`/`on_update` hook that syncs just that one user against the current `enabled_modules`, reusing `sync_blocked_modules()`'s own blocked-list computation logic (would need factoring that computation out of the per-user loop into its own small function first).
- **No hook reverts `block_modules` to fail-open if `Tenant Settings` itself is deleted/trashed.** Deleting the Single record (unusual, but not prevented) would leave every user's `block_modules` at its last-synced state permanently, never reverting to "show everything." Actionable if Tenant Settings deletion ever becomes a real operational path (currently it isn't â€” Tenant Settings is created once and edited, never deleted, per every story's own usage pattern).
- **`sync_blocked_modules()`'s per-user loop is synchronous, non-transactional, and runs on every Tenant Settings save regardless of whether `enabled_modules` actually changed** (e.g. a pure color edit still re-saves every user). For SMB tenant scale (tens of users) this is fine; for a hypothetical large tenant it would block the save request for longer than expected, and a save failure partway through leaves earlier users synced and later ones not, with no rollback. Actionable if/when a real tenant's user count makes this observable â€” the fix is `frappe.enqueue` for the loop plus a short-circuit when `enabled_modules` is unchanged (compare old vs. new in `validate()`/`before_save`, not `on_update`, to have the prior value available).
- **Looping `User.save()` triggers each user's full `on_update()`, including Frappe's own `clear_notifications()` and a `create_contact` background job enqueue â€” an unintended side effect of reusing full-document `.save()` as the sync mechanism.** Every Tenant Settings save (including every preset apply) currently clears every user's notification badges and enqueues N redundant contact-sync jobs. Not incorrect (both are idempotent, harmless operations from Frappe's own perspective), but real, unintended blast radius from a mechanism chosen for simplicity (reusing `.save()`'s normal lifecycle) over precision (a targeted child-table update). Actionable if this ever shows up as a real user complaint ("why did my notifications clear") or queue-depth concern â€” the fix would need a lower-level, non-`.save()`-based update path for just the `block_modules` child table, which carries its own risk of bypassing intended Frappe validation.
- **`enabled_modules` is not constrained to the curated `TOGGLEABLE_MODULES` set at the field, form, or `validate()` level.** An admin can Link any real `Module Def` (e.g. `Setup`, `Core`, a Frappe system module) into it; `sync_blocked_modules()` silently no-ops for anything outside the curated list (only ever iterates `TOGGLEABLE_MODULES` when computing the blocked complement), with no warning that the toggle had no visible effect. Actionable as a small `TenantSettings.validate()` addition (the same seam Story 3.5 already added for dependency warnings) â€” warn (not block, consistent with this epic's own warn-only philosophy) if `enabled_modules` references anything outside the curated set.
- **Concurrent `apply_preset()`/manual-toggle saves for the same tenant have no row-locking** â€” last-write-wins on `enabled_modules`, and interleaved `sync_blocked_modules()` runs could leave users synced against neither the old nor the new state cleanly. Low-probability for a single-admin SMB tenant; actionable if concurrent admin editing ever becomes a real scenario (e.g. multiple ops staff for one tenant) â€” `frappe.db.sql(..., for_update=True)` on the Singles row before read-modify-write.
- **A force-deleted `Module Def` leaves dangling references in `enabled_modules`/preset rows.** Normal (non-`force`) deletion is correctly blocked by Frappe's own link-integrity check (confirmed empirically: `frappe.delete_doc("Module Def", X)` without `force` raises `LinkExistsError` when referenced) â€” only `force=True` (a pattern this project's own `install.py` itself uses elsewhere for Email Account/Website Settings cleanup) bypasses it. Low-probability (Module Def records are ERPNext's own stable module registry, not something an operator would normally delete), but worth a guard (`frappe.db.exists("Module Def", m)` before consuming a row) if it's ever observed.
- **Only 1 of 4 seeded presets (`Retail`) was verified to produce zero self-triggered dependency warnings when applied.** `Services`, `Manufacturing`, and `General` collectively contain every module referenced in `MODULE_DEPENDENCIES` and were never actually run through `apply_preset()` + `validate_dependencies()` and confirmed clean â€” a real verification gap, even though replaying the dependency logic by hand against each preset's module list (done during the Acceptance Auditor's review pass) suggests all 4 are in fact internally consistent. Actionable as a quick follow-up verification pass, not a code change.
- **`MODULE_DEPENDENCIES`'s omission of an `Assets â†’ Accounts` dependency is inconsistent with the reasoning used to include `Selling â†’ Accounts` and `Buying â†’ Accounts`** â€” Asset depreciation posts through `Journal Entry` (an `Accounts`-module doctype), the same "real accounting backing" justification given for the other two. Already documented as a deliberate, conservative choice in Story 3.5's Dev Notes ("not because they have none in a broader business sense"), but worth a second look before considering the dependency map settled â€” a one-line data addition (`"Assets": ["Accounts"]`) if agreed.
- No automated tests exist anywhere in Epic 3's code, consistent with every prior epic's documented decision. All verification (including this review's own patch verification) is one-off manual console/curl sessions.
