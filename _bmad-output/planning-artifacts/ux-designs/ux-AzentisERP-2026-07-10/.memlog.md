- (event) Sources confirmed: prd.md+addendum.md, ARCHITECTURE-SPINE.md, epics.md Epic 1 stories
- (decision) Scope locked: branding/theming layer only (login page, default platform theme, splash screen, tenant brand composition across desk chrome/navbar/footer) -- NOT the inherited ERPNext desk UI (CRM/Sales/Accounting/etc screens), which is explicitly out of scope
- (constraint) AD-3 from architecture spine: branding must render as inline CSS custom properties injected live on every page load -- never a compiled per-tenant static asset file, never requiring bench restart. This is a hard technical constraint on how the DESIGN.md color system must be expressed (CSS variables, not a build step).
- (constraint) AD-2: Tenant Settings fields available for branding: tenant_name, logo, favicon, login_background, primary_color, secondary_color, email_sender_name, email_footer_text, custom_domain -- this is the exact data surface the DESIGN.md token system must map onto (only 2 tenant-controllable colors: primary + secondary)
- (event) UJ-2 from PRD (driving journey): Priya, owns a small retail shop in Pune, never used ERPNext/heard of Frappe, receives welcome email, clicks login link, sees login page styled with her business colors/logo (not generic Frappe login), logs in, sidebar shows only her preset's modules. Climax: nothing reveals Frappe/ERPNext underneath -- feels like software made for her business.
- (event) Epic 1 stories already approved (mechanism-level, no visual decisions yet): 1.1 fork/bootstrap, 1.2 platform brand override (name/logo/favicon/metadata), 1.3 visual theming (colors/login/splash -- currently unspecified WHAT it should look like), 1.4 branded transactional email, 1.5 Tenant Settings data model, 1.6 dynamic per-tenant branding injection (live, no restart)
- (decision) User has no Figma/existing design assets -- design directions to be generated from scratch
- (decision) Both light and dark theme must be supported across the branding layer (login page, splash, desk chrome) -- explicit user requirement
- (decision) Aesthetic direction: premium and modern, optimized for user experience quality -- user delegated specific color/pattern choices, wants to see concrete options rather than describe them verbatim
- (event) Working mode: user said 'go ahead' -- treated as fast-moving but visual-decision-point still warranted (picking a visual personality) since 'premium/modern' is unverifiable without seeing options; generating design-directions renderer output before drafting spines
- (event) 4 design directions rendered to .working/: (1) split-editorial, (2) centered-geometric-minimal, (3) fullbleed-glass, (4) enterprise-trust-panel -- each showing light/dark x platform-default/Priya's-General-Store in a 2x2 grid. Opened in browser for user review.
- (decision) Direction selected: Full-Bleed Glass (animated gradient mesh background, glassmorphic card, subtle drift motion) -- direction-3-fullbleed-glass.html is the reference
- (decision) Scope boundary: full glass/mesh/motion treatment applies ONLY to hero surfaces (login page, splash/loading screen). Post-login desk UI (inherited ERPNext lists/forms/reports) gets a restrained accent-only treatment: navbar tinted with tenant primary_color + logo, no blur/glass/animation -- preserves legibility and performance on dense data screens.
- (decision) Typography: Inter, self-hosted variable font (no external CDN call, keeps app self-contained). Replaces the mockup's system-font stack across all branding-layer surfaces.
- (decision) Splash/loading screen extends the same full-bleed mesh + glass language as login (hero surface), simplified: centered logo mark + loading indicator, no form fields.
- (constraint) Accessibility -- contrast safety: since tenant primary_color/secondary_color are arbitrary user-chosen hex values, the gradient button's fixed white text (--on-primary) could fail contrast against a light tenant color. Rule: server-side, compute button text color (white vs near-black) from the gradient's relative luminance at render time, never hardcode white. All other text (headings, labels, inputs, footnote) renders on the fixed light/dark glass surface tokens, not directly on tenant colors, so it is unaffected by tenant color choice.
- (constraint) Accessibility -- motion: the mesh-blob drift animation must respect prefers-reduced-motion: reduce (pause/disable drift for users who request it) -- was missing from the direction-3 mockup, required for the finalized spine.
- (constraint) Progressive enhancement -- backdrop-filter (glass blur) is not universally supported/performant. Fallback: when unsupported, the glass card renders as a solid/near-opaque surface using the same --glass-bg color at higher opacity, never a broken transparent/unreadable card.
- (decision) Transactional email (FR3) gets light-touch branding only: logo + tenant primary_color accent bar in a simple HTML email header, brand name/footer text swap -- NOT the glass/gradient/mesh treatment, which is both infeasible in email HTML (backdrop-filter/animation unsupported across email clients) and inappropriate for a transactional email register.
- (event) Reconciliation check against PRD/epics Tenant Settings field list found 2 dropped fields: login_background (attach image) and favicon -- neither was addressed in the design directions or spines. Resolving now rather than silently dropping them.
- (decision) login_background resolved: optional per-tenant override. Default (most tenants) = animated CSS mesh gradient from tenant-primary/tenant-secondary. If a tenant uploads a login_background image via Tenant Settings, it replaces the mesh (static image behind the glass card, mesh/blobs suppressed) -- glass card and all other tokens unchanged either way.
- (decision) favicon resolved: trivial pass-through, no design system decision needed -- standard uploaded image used as browser tab icon, wired directly from Tenant Settings.favicon with no DESIGN.md token required.
- (event) Promoted 3 mockups to mockups/: login-full-bleed-glass.html, splash-loading.html, navbar-accent.html. Rejected directions 1/2/4 remain in .working/ as audit trail only, not referenced by either spine. Reviewer gate skipped per user choice (narrow scope, all decisions made directly with user).
- (event) User flagged mobile responsive design needs strengthening -- Responsive & Platform section existed but was thin (2 rows) and no actual phone-viewport mockup was ever rendered, despite Flow 1 (Priya) being explicitly a phone login. Building a real mobile mockup and firming up concrete responsive rules (safe-area insets, 16px input font-size to prevent iOS auto-zoom, touch target sizing, dvh viewport unit, blob scaling).
